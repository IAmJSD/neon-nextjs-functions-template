# Neon Next.js Serverless Template

A full-stack Next.js starter backed by Neon serverless services. It includes a web app, shared API package, Drizzle schema and migrations, Neon Auth, tRPC, Hono REST routes with OpenAPI output, API tokens, public avatar uploads, and a protected MCP server.

## What is included

- Next.js App Router web app with email/password sign-in and sign-up through Neon Auth.
- Neon serverless Postgres access through Drizzle ORM.
- Drizzle schema and migrations for API tokens and MCP OAuth state.
- Hono REST API mounted under `/api`.
- tRPC router shared by server and client components.
- OpenAPI generation for Hono routes.
- API token management UI at `/api-tokens`.
- Profile settings and avatar upload flow backed by the Neon object storage bucket declared in `neon.ts`.
- MCP server with OAuth metadata, dynamic client registration, authorization flow, and an example `get-user` tool.
- Repo-level `neon.ts` policy enabling Neon Auth, an `avatars` bucket, and `api`/`mcp` Neon Functions.

## Repository layout

```text
.
|-- app/packages/web      # Next.js app
|-- app/packages/api      # Hono + tRPC API package
|-- app/packages/mcp      # MCP server and OAuth routes
|-- database              # Drizzle schema, migrations, and DB client
|-- neon.ts               # Neon branch policy: Auth, bucket, API function, MCP function
`-- patches               # patch-package patches applied after npm install
```

## Requirements

- Node.js 24 or newer.
- npm.
- Neon CLI (`neonctl`) for deploying the Neon-managed resources.
- A Neon project with a Postgres connection string.
- Neon Auth configuration for sign-in and session handling.
- Neon object storage credentials if you use avatar uploads.

## Environment variables

Create a root `.env` file for local development. The local dev server and Drizzle config read `.env`, `.env.local`, and `.env.development` from the repository root.

Some values below are generated by Neon and injected into deployed Neon Functions. For local development, either put equivalent values in `.env` or run your dev command through `neon-env` so local processes receive the same Neon-managed values.

```bash
# Neon-managed values. These are injected in Neon runtimes, but useful in .env locally.
DATABASE_URL=
NEON_AUTH_BASE_URL=
NEON_AUTH_JWKS_URL=
AWS_ACCESS_KEY_ID=
AWS_SECRET_ACCESS_KEY=
AWS_ENDPOINT_URL_S3=
AWS_REGION=

# App-managed values. You provide these.
NEON_AUTH_COOKIE_SECRET=
NEXT_PUBLIC_API_URL=
MCP_WEB_ORIGIN=
MCP_RESOURCE_SERVER_URL=
```

Variables this template reads:

| Variable | Used by | Notes |
| --- | --- | --- |
| `DATABASE_URL` | Database, API, MCP, Drizzle | Neon Postgres connection string. |
| `NEON_AUTH_BASE_URL` | Web, MCP | Base URL for Neon Auth. |
| `NEON_AUTH_COOKIE_SECRET` | Web, API | Signs and verifies Neon Auth session cookies. |
| `NEON_AUTH_JWKS_URL` | MCP | Verifies OAuth bearer tokens. |
| `NEXT_PUBLIC_API_URL` | Web in production | Base URL for the deployed API function. Local development uses `/api/rpc`. |
| `MCP_WEB_ORIGIN` | MCP function | Public origin for the web app, used for browser authorization redirects. |
| `MCP_RESOURCE_SERVER_URL` | MCP function | Public URL for the deployed MCP resource server. |
| `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_ENDPOINT_URL_S3`, `AWS_REGION` | API avatar uploads | S3-compatible credentials for the `avatars` bucket declared in `neon.ts`. |

Neon-managed values:

| Variable | Source | Notes |
| --- | --- | --- |
| `NEON_BRANCH` | Neon branch runtime | Injected by Neon Functions and produced by `neon-env`; this template does not read it directly. |
| `DATABASE_URL` | Neon Postgres | Injected for the selected branch. Required locally unless you use `neon-env run`. |
| `DATABASE_URL_UNPOOLED` | Neon Postgres | Injected for the selected branch. The current code uses `DATABASE_URL`, but this is available for direct, unpooled connections. |
| `NEON_AUTH_BASE_URL` | Neon Auth | Available because `neon.ts` enables Neon Auth. |
| `NEON_AUTH_JWKS_URL` | Neon Auth | Available because `neon.ts` enables Neon Auth; used by the MCP package to verify bearer tokens. |
| `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_ENDPOINT_URL_S3`, `AWS_REGION` | Neon object storage | Available for the `avatars` bucket declared in `neon.ts`; used by the avatar upload tRPC procedures. |

App-managed values:

| Variable | Source | Notes |
| --- | --- | --- |
| `NEON_AUTH_COOKIE_SECRET` | You provide | Used by the web app to sign session cookies and by the API to verify session-backed requests. `neon.ts` passes it through to the deployed API function. |
| `NEXT_PUBLIC_API_URL` | You provide | Public URL of the deployed API function. Only required by the web app outside local development. |
| `MCP_WEB_ORIGIN` | You provide | Public origin of the web app. `neon.ts` passes it through to the deployed MCP function. |
| `MCP_RESOURCE_SERVER_URL` | You provide | Public URL of the deployed MCP function. `neon.ts` passes it through to the deployed MCP function. |

Optional variables supported by the code include `NEON_AUTH_COOKIE_DOMAIN`, `NEON_AUTH_COOKIE_SAME_SITE`, `MCP_RESOURCE_NAME`, `MCP_SERVICE_DOCUMENTATION_URL`, `MCP_OAUTH_SCOPES`, `MCP_OAUTH_ISSUER`, `MCP_OAUTH_AUTHORIZATION_URL`, `MCP_WEB_AUTHORIZATION_URL`, and `MCP_OAUTH_SIGN_IN_URL`.

## Getting started

Install dependencies:

```bash
npm install
```

Apply the existing database migrations:

```bash
npm run drizzle-kit -- migrate
```

Start the development server:

```bash
npm run dev
```

Open `http://localhost:3000`.

In development, the Next.js app imports the API and MCP packages directly and serves them through local routes. In production, the web app expects the API and MCP endpoints to be deployed as separate Neon Functions and configured through the environment variables above.

## Common commands

```bash
# Start Next.js in development mode
npm run dev

# Generate a new Drizzle migration after changing database/schema.ts
npm run drizzle-kit -- generate

# Apply pending migrations
npm run drizzle-kit -- migrate

# Open Drizzle Studio
npm run drizzle-kit -- studio

# Deploy the Neon-managed pieces from neon.ts
neonctl deploy --project-id <project-id> --branch <branch-id-or-name> --env .env

# Build the web app
npm run web:build

# Start the built web app
npm run web:start
```

## Routes

| Route | Description |
| --- | --- |
| `/` | Auth-protected home page with client and server tRPC examples. |
| `/auth/sign-in` | Email/password sign-in. |
| `/auth/sign-up` | Email/password sign-up. |
| `/settings` | Profile details and avatar upload. |
| `/api-tokens` | Create and revoke bearer API tokens. |
| `/api/v1/hello` | Example authenticated Hono REST endpoint. |
| `/api/rpc/:path` | tRPC endpoint. |
| `/openapi` | OpenAPI output for local development. Use the deployed API function URL in production. |
| `/mcp` | Local MCP endpoint in development. In production, use the deployed MCP function URL. |
| `/mcp/authorize` | Web-side MCP OAuth authorization redirect. |

REST API calls accept `Authorization: Bearer <api-token>` for tokens created in `/api-tokens`. The API also supports `Authorization: Session <session-token>` for session-backed calls.

## Where to customize

- Rename the app in `app/packages/web/app/navbar.tsx`.
- Update the landing page in `app/packages/web/app/page.tsx`.
- Add Next.js pages and route handlers under `app/packages/web/app`.
- Add tRPC procedures in `app/packages/api/src/trpc/index.ts`.
- Add REST/OpenAPI routes in `app/packages/api/src/apiRouters`.
- Add MCP tools in `app/packages/mcp/src/mcpServer.ts`.
- Change database tables in `database/schema.ts`, then generate a migration.
- Change Neon Auth, bucket, and function policy in `neon.ts`.
- Search for `TODO` comments and replace the starter values before deploying.

## Database workflow

The Drizzle schema lives in `database/schema.ts`. Existing migrations create:

- `api_tokens` for bearer token access.
- `mcp_oauth_clients` for MCP dynamic client registration.
- `mcp_oauth_authorization_codes` for MCP OAuth authorization code flow.

After schema changes:

```bash
npm run drizzle-kit -- generate
npm run drizzle-kit -- migrate
```

## Deployment notes

`neon.ts` declares the Neon resources this template expects:

- Neon Auth enabled.
- Public-read `avatars` bucket.
- `api` Neon Function from `app/packages/api/src/index.ts`.
- `mcp` Neon Function from `app/packages/mcp/src/index.ts`.

Run `neonctl deploy` from the repository root to split out and deploy the Neon-managed pieces declared in `neon.ts`:

```bash
neonctl auth
neonctl deploy --project-id <project-id> --branch <branch-id-or-name> --env .env
```

The `--env .env` flag loads app-managed values before evaluating `neon.ts`, so the deployed functions receive values such as `NEON_AUTH_COOKIE_SECRET`, `MCP_WEB_ORIGIN`, and `MCP_RESOURCE_SERVER_URL`. By default, `neonctl deploy` also pulls Neon-managed branch env vars back into local `.env` after a successful apply; pass `--no-env-pull` if you prefer to inject them at runtime with `neon-env run` or `neonctl dev`.

Use `--update-existing` when you want to accept updates to existing remote settings, and `--allow-protected` when deploying to a protected Neon branch.

After the Neon deploy, deploy the Next.js web app separately. Configure the web app with the public API function URL in `NEXT_PUBLIC_API_URL`, configure the MCP function with `MCP_WEB_ORIGIN` and `MCP_RESOURCE_SERVER_URL`, and make sure the database migrations have run against the target branch.

Before shipping, replace starter defaults and TODOs such as the navbar title, page metadata, OpenAPI title, and MCP server name/description.
